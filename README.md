# 프로젝트명

건설 도면 탐색 플랫폼

## 실행 방법

npm install
npm run dev

## 기술 스택

- React 19.2.0
- TypeScript 5.9.3
- Vite 7.3.1

## 구현 기능

- [x] 도면 탐색: 사용자가 원하는 도면을 왼쪽 사이드바에서 찾을 수 있습니다.
- [x] 도면 표시: 선택한 도면 이미지를 중앙 화면에 표시합니다.
- [x] 컨텍스트 인식: 현재 어떤 도면을 보고 있는지, 헤더 및 이미지 표시 부분에 도면 이름이 띄워집니다.
- [x] 도면 확대: 도면 비교 시 마우스 더블클릭으로 1.5배 확대하여 자세히 도면 관찰 가능합니다.
- [x] 도면 비교: 우측 상단 "도면 비교"를 누르면 최대 4개까지 도면을 한 화면에 띄워 비교할 수 있습니다.
- [x] 도면 변경이력 추적: 오른쪽 상단 사이드바에 각 도면에 대한 설명과 변경 내용이 띄워집니다. 리비전이 있는 경우 순서대로 함께 띄워집니다.

## 미완성 기능

- [] 도면 겹치기: 도면 2개를 완전히 겹쳐 비교하는 기능은 구현하지 못했으며, 이 대체로 행 변환을 통해 도면 여러 개가 한 눈에 보일 수 있도록 구현했습니다.
- [] 도면 변경이력 갱신: 다른 주제의 도면을 비교할 때, 오른쪽 사이드바 변경 이력이 대표적인 것만 띄워지는 문제를 파악하였으나, 수정하지 못했습니다.
- [] 메타 데이터 폴리곤 활용: 각 도면의 좌표 정보를 활용하지 못했습니다.

## 설계 방식

- **데이터 분석**:
  - `metadata.json`은 도면(Drawing), 공종(Discipline), 영역(Region), 그리고 개정(Revision)이 계층적으로 얽힌 복합적인 구조로 되어 있습니다. 도면 좌표 배치도가 포함되어 있으며 각 건물이 어디에 속하는지 표시되는 것으로 파악했습니다.
  - **더 나은 데이터 표현 방법**: 향후 직접 데이터베이스 연동을 통해 쉽게 쿼리로 찾을 수 있도록 하는 방식이 더 나을 것이라 생각됩니다.

- **접근 방식**:
  1. **데이터 모델링**: `metadata.json`을 분석하고 인터페이스를 구성하고 데이터의 계층 구조를 명확히 했습니다.
  2. **데이터 추출**: 많고 다양한 데이터 중, 화면에 띄워질 중요한 데이터만 추출하여 사용자 화면에 표시했습니다.
  3. **레이아웃 구성**: 헤더 영역을 제외하고 화면을 3분할하여 각 화면 별 기능을 나누었습니다. (도면 선택, 도면 표시, 도면 설명)
  4. **기능 구현**: 왼쪽 사이드바에는 공종 선택과 해당 공종에 있는 도면 목록이 띄워지도록 구현했으며, 중앙부에는 선택한 도면이 표시, 오른쪽 사이드바에는 표시된 도면에 대한 변경 이력을 표시하는 기능을 구현했습니다. 또한 우측 상단에 도면 비교 버튼을 통해서 최대 4개까지 원하는 도면을 선택하여 한 눈에 비교할 수 있도록 구성했습니다.

- **UI 설계 결정**:
  - **레이아웃**: 공종 분야와 도면을 선택할 수 있는 왼쪽 사이드바, 중앙 도면 표시, 도면 변경 이력이 표시되는 우측 사이드바로 구성했습니다. 또한 상단 헤더에 현재 선택된 분야와 도면 이름이 표시됩니다.
    이는 사용자가 한 화면에서 도면 선택부터 변경 이력까지 모두를 한 눈에 파악할 수 있는 구조로 구상했습니다.

  - **고려했던 대안 및 장단점**: 도면 비교 시 도면끼리 완전히 겹쳐서 투명율을 조절하여 비교된 도면이 무엇인지 파악하려 했으나, 도면이 겹쳤을 때 어떤 도면이 바뀐 것인지에 대한 기준이 모호하고 사용자가 파악하기 오히려 어려울 것이라고 판단하여, 한 눈에 파악할 수 있는 2행 2열 구조로 배치했습니다.

- **기술 선택**:
  - **React & TypeScript**: 컴포넌트 기반의 UI 개발과 엄격한 타입 체크를 통해 데이터 구조의 복잡성을 제어하고자 선택했습니다.
  - **State Management**: React 내장 Hooks(`useState`, `useMemo`)를 적극 활용하여 효율적인 데이터 흐름을 구현했습니다.

- **어려웠던 점 및 개선 방안**:
  - **어려웠던 점**: <br>
    도면 비교 모드를 만드는 과정에서 도면을 몇 개까지 띄워야 할지, 또 도면 배치를 어떻게 해야 사용자 경험을 만족시킬 수 있을 지에 관한 부분이 까다로웠습니다. 직접 A4용지에 손으로 그려가며 사용자 화면에서 어떤 배치가 한 눈에 들어올지, 또는 어떤 기능이 필수적일지 고민을 많이 했습니다.
    <br/><br/>
  - **개선 방안 및 추가하고 싶은 기능**:
  1. 메타데이터의 좌표 정보를 활용해 도면 내 특정 영역(Region)을 강조하는 기능을 추가하고 싶습니다.
  2. 오른쪽 사이드바 변경 이력에서 즉석으로 메모할 수 있는 메모장 기능을 추가하고 싶습니다. 메모장에 사용자가 도면을 보고 즉각적으로 메모할 사항이나 회의 등을 기록할 수 있는 기능을 추가한다면 시나리오에서 적힌대로 회의 시 변경사항 추적 및 설계 도면 파악을 빠르게 할 수 있을 것입니다.
  3. 도면 편집 기능을 추가하여 회의 시 도면 사진 위에 펜으로 직접 그리면서 표시할 수 있는 기능을 추가하고 싶습니다. 이는 회의 시 도면에 어떤 점이 바뀌었는지 리비전을 띄워두고 직접 체크표시하며 더 빠르고 효율적이게 판단을 할 수 있을 거라 생각됩니다.

- **문제를 풀이하는 나만의 방식**:
  - 과제를 수행할 때는 우선 어떤 대상으로 이 결과물이 만들어지는지를 파악합니다. 그리고 해당 대상이 나라고 생각하고 직접 화면을 손으로 그리면서 설계해보고 시나리오가 있다면 해당 내용을 분석하여 어떤 기능이 필수적일지, 어떤 기능이 유용할지 먼저 쭉 나열하는 형식으로 과제의 전반적인 흐름을 공부합니다.
  - 항상 사용자 관점에서 앱 및 웹을 설계해온 저는 불필요한 기능은 최대한 줄이고, 유용한 기능은 최대한 편리하게 이용할 수 있도록 쉽고 직관적이게 구현하려고 노력했습니다.
  - 짧은 시간동안의 과제였지만 사용자 화면에서 한 눈에 들어오는 양쪽 사이드바 배치, 그리고 정중앙의 너비를 채우는 도면 표시 등 간결하고 사용자에게 필요한 기능을 담은 화면을 설계했습니다.
  - 도면 비교 시, 완전 겹치는 것 보다 화면에 사용자가 선택한 도면을 전부 다 띄우는 방식을 택한 이유는 이미 이 도면 탐색 플랫폼을 사용하는 사용자는 전반적인 도면 배치에 대한 이해도가 있는 사람일 것이고, 옆으로 나란히 배치하고 한 눈에 띄우는 구조가 더 쉽게 도면 개정을 파악할 수 있을 것이라 생각했습니다. 심지어 우측 사이드바에는 리비전을 포함하여 변경 이력을 모두 다 표시해뒀기 때문에 더더욱 용이할 것으로 판단했습니다.
  - 향후 더 많은 데이터가 주어진다면 추가하고 싶은 기능들이 많습니다. 메모장 표시, 도면 오버레이하여 변경 추적, 좌표 사용 등 다양한 기능을 추가한다면 해당 플랫폼이 빠르게 성장할 수 있을 것으로 판단됩니다.
